# 从被动响应到自进化：Agent 技术范式演化演讲稿

> 用途：配合 `agent-evolution-ppt.html` 使用的 1 小时演讲稿。  
> 建议节奏：正式讲解约 50-55 分钟，预留 5-10 分钟互动、提问或临场扩展。  
> 使用方式：每页都包含“主讲稿”“可展开例子”“过渡句”。如果现场时间紧，只讲主讲稿；如果需要撑满时间，就把例子和补充段落展开。

## 整体时间分配

- 开场与问题建立：8 分钟
- Agent 四个阶段：18 分钟
- 六个核心技术概念：25 分钟
- 选型建议与总结：8 分钟
- 互动与提问：5 分钟

## 讲述主线

这场分享不要讲成“Agent 框架盘点”，而要讲成一个演化故事：

1. 早期大家以为 Agent 是“会调用工具的 Chatbot”。
2. 后来发现模型不稳定，所以用 Workflow 给它铺路。
3. 再后来模型推理能力上来，Agent 开始能自己规划、自己执行、自己校验。
4. 现在更进一步，Agent 不只是完成任务，还开始沉淀经验，变成一个会成长的系统。

核心观点可以反复强调：

> Agent 的变化，不是“又多了一个新框架”，而是工程范式变了。  
> 我们越来越清楚：要用工程化确定性，承载模型的不确定性。

---

## 第 1 页：从被动响应到自进化

预计时间：3 分钟

### 主讲稿

大家好，今天想和大家聊一个最近两三年变化非常快的话题：Agent。

如果我们把时间拉回到 2023 年，很多人第一次听到 Agent 的时候，脑子里想象的可能还是一个“增强版聊天机器人”：我问它一个问题，它想一想，可能查一下工具，然后回答我。但到了今天，尤其是 Claude Code、Codex、Manus、OpenClaw、Hermes 这些产品和框架出现之后，Agent 已经不太像一个只会回答问题的系统了，它更像一个能执行任务的数字员工。

所以今天的主题叫“从被动响应到自进化”。这几个字其实对应的是 Agent 的三次角色变化：

第一，它从“回答者”变成了“执行者”。以前它主要负责给建议，现在它开始真正改代码、查资料、跑命令、生成文件。

第二，它从“单次工具”变成了“工作系统”。以前你用完一次就结束，现在它需要工作空间、记忆、技能、规则、上下文管理。

第三，它从“静态能力”开始走向“可积累能力”。也就是说，它不只是完成任务，还能把经验沉淀下来，下次做得更好。

今天这套 PPT 不想把门槛讲得太高。我会尽量用类比和例子来讲清楚：Agent 到底怎么变了？为什么会这么变？我们做技术选型的时候又该怎么判断？

### 可展开例子

可以先用一个普通人的例子开场：

假设你要让一个助理帮你办一场线下活动。

最早的 Chatbot 像什么？像你问它：“办活动要注意什么？”它会列一个清单，比如场地、嘉宾、海报、报名表、物料。这个阶段它主要是给建议。

早期 Agent 往前走了一步。你说：“帮我生成一份活动方案。”它可能调用搜索工具、调用文档生成工具，给你写一个方案。这已经不只是聊天了，但它仍然需要你一步步推动。

Workflow Agent 像一个 SOP 系统。你把流程固定好：第一步收集信息，第二步生成海报文案，第三步发报名链接，第四步提醒参会者。它能稳定执行，但灵活性有限。

自主 Agent 更像一个真正的助理。你只说：“下个月帮我办一场 Agent 技术分享会，目标 100 人报名。”它会自己拆任务、查场地、写宣传文案、排日程、检查遗漏。

自进化 Agent 则更进一步：这次活动办完以后，它会总结：“上次报名转化率最高的是这个标题；提醒短信提前 1 天发效果更好；现场签到容易拥堵，下次要多放一个二维码。”这就是经验沉淀。

### 过渡句

所以今天我们不是简单介绍几个新工具，而是看清楚 Agent 为什么会从“会聊天”走向“会做事”，再走向“会成长”。

## 第 2 页：为什么大家越学 Agent，反而越容易迷糊？

预计时间：4 分钟

### 主讲稿

我自己最近有一个很强烈的感受：很多同学不是不学习 Agent，而是学得太多、太杂，最后反而更迷糊。

原因有三个。

第一个原因是新产品出现得太快。今天大家刚听说 Claude Code，明天又有 Codex，后天又出现 OpenClaw、Hermes。每个产品都会带一套新概念：Skills、Memory、Sandbox、Harness、Sub-agent、Agent Loop。你如果只追概念，很快就会被概念淹没。

第二个原因是旧文章还在流传。现在网上很多 Agent 入门文章，其实仍然停留在 2023 年那套范式：LLM 加 Planning，加 Tools，加 Memory，再画一个 ReAct 循环。这些内容并不是错的，但是它们解释不了今天很多 Agent 产品的真实工作方式。

第三个原因是新旧概念混在一起。比如有人还在把 Function Call 当成 Agent 工具调用的唯一方式，但现在很多 Coding Agent 更依赖 CLI 和脚本；有人还在把 Memory 简单等同于向量数据库，但现在很多实践已经转向文件系统、Markdown、日志和轻量检索的混合模式。

所以今天我们要做的事情，就是把这些概念放回演化脉络里看。很多技术不是突然冒出来的，也不是谁完全替代谁，而是在不同阶段解决不同的问题。

### 可展开例子

可以举一个“手机发展史”的类比：

如果一个人今天学手机技术，但资料混在一起，有人给他讲功能机，有人讲智能机，有人讲折叠屏，有人讲可穿戴设备，他一定会乱。

功能机时代，最重要的是打电话、发短信、电池耐用。智能机时代，核心变成了 App 生态、触屏交互、移动互联网。到了今天，又开始强调 AI 助手、端侧模型、多设备协同。

这些东西不是简单替代关系。功能机的“通信稳定性”今天依然重要；智能机的 App 生态依然重要；AI 助手是新的增量能力。Agent 也是一样。早期 ReAct、Workflow、自主 Agent、自进化 Agent 都有价值，关键是别把它们混成一团。

再举一个企业落地的例子：

你要做一个报销审核 Agent。这个场景规则非常明确，发票金额、审批层级、预算科目都有硬规则。你如果非要用一个完全自主 Agent，让它自由规划审批逻辑，反而可能不稳定。这里 Workflow 就很好。

但如果你要做一个“帮我分析这个客户为什么流失，并给出挽回方案”的 Agent，这就不是固定流程能完全解决的，因为它要看客户历史、看沟通记录、看竞品动态、生成策略。这个时候自主 Agent 更有用。

### 过渡句

所以我们先不要急着问“哪个 Agent 框架最好”，而要先问：Agent 到底经历了哪些阶段？

---

## 第 3 页：不是追热点，而是看懂变化背后的逻辑

预计时间：3 分钟

### 主讲稿

今天的分享目标不是让大家记住一堆新名词。

因为名词会变得很快。今天叫 Agent Skills，明天可能叫 Capabilities，后天又叫 Playbooks。但底层问题不会变：模型不稳定怎么办？上下文不够怎么办？工具怎么调用？长期任务怎么不中断？做完一次任务以后，经验怎么复用？

我希望大家听完之后，能建立一个判断框架：

当你看到一个新的 Agent 框架时，你可以不被营销话术带着走，而是拆开看：

它的 Prompt 是怎么组织的？  
它的 Planning 是模型自己做，还是外部流程控制？  
它的 Memory 是向量库，还是文件系统，还是两者结合？  
它的 Tools 是 API，还是 CLI，还是脚本？  
它有没有 Runtime？有没有隔离？有没有持久化工作空间？  
它能不能把经验沉淀下来？

只要你能用这几个问题去拆解，很多新框架就没那么神秘了。

### 可展开例子

可以举“看车”的例子：

普通人看车，可能看外观、品牌、屏幕大不大。懂车的人会看发动机、底盘、变速箱、电池、智能驾驶、安全结构。

看 Agent 也是一样。外面包装可能都叫“全自动智能体”“下一代 AI 员工”，但我们要看底层结构：

有没有任务规划？  
有没有工具执行？  
有没有错误恢复？  
有没有权限边界？  
有没有日志和可追踪性？  
有没有记忆沉淀？

这样一拆，就知道它到底是 Demo、工具、框架，还是可以上生产的系统。

### 过渡句

接下来我们先从宏观上看：过去三年，Agent 大致走过了哪四个阶段。

---

## 第 4 页：Agent 发展的四个阶段

预计时间：5 分钟

### 主讲稿

我把 2023 到 2026 这几年 Agent 的发展，粗略分成四个阶段。

第一阶段是 2023 年的早期 Agent，也就是被动式 ReAct。它的特点是“一问一答”，模型根据用户输入进行推理，必要时调用工具，然后返回结果。它很重要，因为它定义了 Agent 的基本想象力，但能力边界比较短。

第二阶段是 2024 年的工作流 Agent。这个阶段，大家发现单纯靠模型自由推理不稳定，于是开始用 Workflow 把流程固定下来。它不是最酷的，但非常适合企业落地，因为稳定、可控、可解释。

第三阶段是 2025 年的自主 Agent。随着模型推理能力变强，Agent 开始可以做更复杂的 Planning，可以处理长程任务。像 AI Coding Agent 就是典型代表，它不是只回答“代码怎么改”，而是能自己读代码、改代码、跑测试、修错误。

第四阶段是 2026 年开始变得更明显的自进化 Agent。它的核心不是做一次任务，而是做完之后能留下经验，沉淀 Skill、Memory、知识库，甚至通过反馈继续优化模型或策略。

这四个阶段不是线性淘汰关系，而是能力层层叠加。就像交通工具一样，有人需要自行车，有人需要地铁，有人需要飞机。不是飞机出现后，自行车就没有价值，而是不同场景选择不同工具。

### 可展开例子

可以用“做一份行业研究报告”贯穿四个阶段：

早期 Agent：你问它“帮我分析一下 Agent 行业发展趋势”，它搜索几篇文章，总结一下，给你一份回答。它像一个会搜索的助手。

Workflow Agent：你事先设计好流程：搜索资料、筛选来源、提取观点、生成大纲、写正文、生成图表、输出 PDF。每一步都固定，模型只在节点里发挥。这样做报告质量稳定，但如果中途发现新问题，它不一定会主动调整。

自主 Agent：你只给目标：“我要一份面向管理层的 Agent 行业报告，重点看技术趋势、商业化和风险。”它会自己拆任务：先找资料，再分类，再对比，再写初稿，再检查逻辑，再补案例。这时候它像一个研究助理。

自进化 Agent：它做完这份报告后，会记录：哪些数据源可靠，哪些搜索关键词有效，管理层更喜欢什么结构，上次哪些部分被用户改得最多。下次再写报告，它就能直接复用这些经验。

### 过渡句

我们先逐个拆一下这四个阶段，看看每个阶段到底解决了什么问题，又留下了什么问题。

## 第 5 页：阶段一，早期 Agent：增强版 Chatbot

预计时间：4 分钟

### 主讲稿

早期 Agent 最典型的关键词就是 ReAct：Reasoning 加 Acting。

通俗讲，就是模型先想一下，然后做一个动作，再观察结果，再决定下一步。这个思想很自然，也很有启发性。比如你让 Agent 查天气，它会先判断需要调用天气 API，然后拿到结果，再组织语言回答你。

但这个阶段的 Agent 仍然很像增强版 Chatbot。它的交互方式还是用户推一下，它走一步。你问一个问题，它回答一个问题；你下一个指令，它再执行一次。

它能做的事情主要是短链路任务，比如查询资料、生成一段文案、调用一个工具、简单总结。只要任务超过几轮推理，它就容易偏离。

为什么？因为当时基础模型的推理能力、上下文长度、工具调用稳定性都不够。你让它做三步以内还可以，做十步、二十步就很容易忘记目标，或者走到一半开始胡编。

### 可展开例子

可以举一个“让早期 Agent 帮你订旅行计划”的例子：

你说：“帮我安排一个三天两晚的杭州旅行。”  
它可能能给你一个不错的行程：第一天西湖，第二天灵隐寺，第三天良渚。

但如果你继续要求：“我带老人和小孩，预算 3000，不能走太多路，酒店要靠近地铁，第二天下午还要安排一个商务会面，晚上要吃不辣的餐厅，帮我考虑天气变化，并给出交通路线。”

这时候早期 Agent 很容易崩。它可能前面记得老人小孩，后面忘记预算；前面说不走太多路，后面安排一天 2 万步；前面说不辣，后面推荐火锅。

这不是因为 ReAct 思想错了，而是模型能力和工程环境还没跟上。

也可以举代码场景：

早期 Agent 能回答：“这个函数可能有什么问题？”  
但你让它在一个大型代码库里定位 bug、修改多个文件、跑测试、修复测试失败，它就很难稳定完成。

### 过渡句

所以很快大家发现：如果模型自己走不稳，那能不能先把路修好？这就进入了第二阶段，工作流 Agent。

---

## 第 6 页：阶段二，工作流 Agent：先把路修好，再让模型跑

预计时间：5 分钟

### 主讲稿

工作流 Agent 的核心思想非常工程化：既然模型不稳定，那就不要把所有决策都交给模型。我们把流程设计好，把关键节点固定住，让模型只在合适的位置发挥。

这其实非常符合企业软件的思路。企业不是追求“看起来很智能”，而是追求“每天稳定把事情做完”。

比如一个客服工单处理流程，可以拆成：识别问题类型、查询用户信息、检索知识库、生成回复、判断是否需要转人工、记录处理结果。这里每一步都可以固定，模型只负责分类、摘要、生成话术。

这种方式的好处是稳定、可控、可解释。出了问题，你知道是哪一步出了问题。流程也容易审计，适合 to B 场景。

但它的缺点也很明显：灵活性不足。如果用户的问题不在预设流程里，或者中途出现异常，Workflow 往往不会主动创造新路径。

### 可展开例子

可以讲一个“流水线工厂”的类比：

Workflow Agent 像工厂流水线。每个工位负责一件事：切割、打磨、组装、质检、包装。只要产品类型稳定，流水线效率极高，质量也稳定。

但如果今天客户突然拿来一个完全不同的定制产品，流水线就麻烦了。因为它不是为了开放式探索设计的。

再讲一个企业例子：

假设银行做贷款审批。这个场景非常适合 Workflow。因为审批规则、风险模型、合规要求都必须清楚。你不能让 Agent 自由发挥：“我觉得这个人挺靠谱，给他放款吧。”这显然不可接受。

但如果是“分析这个行业的潜在客户名单，并找出最值得拜访的 20 家企业”，这个任务就更开放，需要搜索、判断、比较、反复调整，纯 Workflow 会显得僵硬。

### 可展开观点

这里可以强调一个很重要的判断：

Workflow 不是落后技术。很多时候它是最适合生产的技术。

现在很多人一听到自主 Agent，就觉得 Workflow 过时了，这是不对的。Workflow 的价值在于保证下限。只要业务规则明确、流程稳定、容错率低，Workflow 仍然是非常好的选择。

### 过渡句

但随着基础模型变强，Agent 开始不满足于只在固定轨道里跑。它开始能够自己拆任务、自己走路径，这就是第三阶段。

---

## 第 7 页：阶段三，自主 Agent：从“帮你答”到“帮你做”

预计时间：5 分钟

### 主讲稿

自主 Agent 最大的变化，是它开始从“回答问题”变成“完成任务”。

这两者差别非常大。

回答问题通常是一次性的。你问：“这段代码为什么报错？”模型解释给你听。

完成任务则是一个过程。你说：“帮我修复这个 bug。”Agent 需要读错误日志、定位代码、理解业务、修改文件、运行测试、看到测试失败后继续修、最后告诉你改了什么。

这个过程里最重要的是 Planning。Agent 要能把一个模糊目标拆成可执行步骤，而且执行过程中还能动态调整。

AI Coding Agent 是最典型的例子。它面对的不是一个孤立问题，而是一个真实代码库。代码库里有历史包袱、有命名规范、有测试、有 CI、有用户未提交的改动。Agent 必须像一个工程师一样工作，而不是像一个问答机器人一样回答。

### 可展开例子

可以讲一个真实工程任务：

用户说：“这个上传功能有问题，帮我修一下。”

普通 Chatbot 可能会说：“你可以检查前端表单、后端接口、文件大小限制、网络请求。”

自主 Agent 会怎么做？

第一步，它先看错误现象，可能读取日志或复现问题。  
第二步，它搜索上传相关代码。  
第三步，它理解前端如何发请求，后端如何接收文件。  
第四步，它找到可能的问题，比如字段名不一致、Content-Type 错误、文件大小限制没提示。  
第五步，它修改代码。  
第六步，它跑轻量检查或测试。  
第七步，如果失败，继续修。  
第八步，最后总结改动。

这个过程已经很像初级工程师了。

再举一个非代码场景：

你让 Agent “帮我准备一个客户拜访方案”。它需要查客户公司、看行业动态、整理痛点、生成沟通提纲、准备演示材料、列出可能被问到的问题。这也是一个长程任务。

### 可展开观点

自主 Agent 的核心不是“更会聊天”，而是“更会闭环”。

闭环包括：目标理解、任务拆解、工具执行、结果观察、错误修正、最终交付。

如果一个系统只能生成计划，但不能执行，它不是完整 Agent。  
如果一个系统能执行，但不能观察和修正，它也很脆弱。  
真正有用的 Agent 必须能在不确定环境里持续推进任务。

### 过渡句

但做到自主执行以后，又出现了一个新问题：每次做完都归零，很浪费。于是下一步就是让 Agent 把经验留下来。

---

## 第 8 页：阶段四，自进化 Agent：做完任务，还把经验留下来

预计时间：5 分钟

### 主讲稿

自进化 Agent 解决的是“静态模型”和“动态世界”的矛盾。

模型训练完以后，它的参数基本是静态的。但真实世界每天都在变化：你的项目代码在变，团队规范在变，用户偏好在变，业务流程在变。如果 Agent 每次都只靠模型原始能力，那它永远需要你重复告诉它同样的事情。

自进化的核心，就是把经验沉淀下来。

比如这次你告诉它：“我们团队提交代码前只做轻量语法检查，不跑完整环境验证。”如果它能把这个偏好记录下来，下次就不用你再说。

比如它做某类任务时发现一个稳定流程：先读 SPEC，再改代码，再检查 linter，再总结部署影响。它可以把这个流程沉淀成 Skill。

比如它在项目里理解了一套领域知识：什么叫合同模板、什么叫字段识别、什么叫钉盘集成。它可以把这些沉淀到项目知识库里。

这就是“越用越好用”的来源。

### 可展开例子

可以举“新人变老员工”的类比：

一个刚入职的新同事，即使很聪明，也需要你反复告诉他：项目怎么启动、代码规范是什么、哪些接口不能碰、上线流程是什么、谁负责什么模块。

但一个老员工不一样。他做过几次以后，会形成经验。他知道哪些地方容易出 bug，知道哪个接口历史上改过，知道测试环境有个坑，知道某个需求背后真正的业务目标。

自进化 Agent 的目标，就是让 Agent 从“每次都像新人”变成“越来越像老员工”。

再举个人助手例子：

你让 Agent 帮你写文章。第一次它不知道你的风格，可能写得很官方。你改了几次以后，如果它能记住：你喜欢先抛问题，再讲演化，再讲方法论；你不喜欢太营销的话；你喜欢技术概念配生活类比。下次它写文章就会更接近你。

### 风险补充

这里也要提醒，自进化不是没有风险。

如果 Agent 沉淀了错误经验，它下次可能更稳定地犯错。  
如果 Memory 没有边界，它可能记住不该记的信息。  
如果 Skill 没有维护，旧规则可能和新需求冲突。  

所以自进化不是“随便记忆”，而是需要可读、可改、可审查的沉淀机制。文件系统、Markdown、日志、版本管理在这里就变得很重要。

### 过渡句

讲完四个阶段，我们要特别强调一点：这些阶段不是互相淘汰，而是可以组合成工具箱。

---

## 第 9 页：四个阶段不是替代关系，而是工具箱关系

预计时间：4 分钟

### 主讲稿

很多技术讨论容易陷入一个误区：新东西出来以后，旧东西是不是就没用了？

在 Agent 这里，答案是否定的。

ReAct 依然有价值，因为很多短任务就是需要推理加工具调用。  
Workflow 依然有价值，因为很多企业流程就是需要稳定可控。  
自主 Agent 很有价值，因为复杂任务需要灵活规划。  
自进化 Agent 也很有价值，因为长期使用需要经验沉淀。

真正成熟的系统，往往是混合的。

比如一个企业级数据分析 Agent，外层可能是自主 Agent：用户问一个开放问题，它自己判断要查哪些数据、生成哪些图表。但涉及权限校验、数据脱敏、审批发布时，必须走固定 Workflow。做完以后，它还要把分析模板、常用指标、用户偏好沉淀到 Memory 或 Skill。

所以我们要从“选一个最先进的范式”，转向“为不同任务选择合适的范式”。

### 可展开例子

可以用医院类比：

挂号缴费适合 Workflow。流程固定，不能自由发挥。  
医生诊断更像自主 Agent。需要综合症状、检查结果、病史，动态判断。  
医院知识库和病例积累更像 Memory。经验越多，诊断越准。  
医学指南和操作规范更像 Skill。它告诉医生某类问题应该按什么路径处理。

没有一个范式能解决所有问题。医院需要挂号系统，也需要医生判断，也需要病例库，也需要医学指南。

### 过渡句

接下来我们进入更细的部分：从六个核心模块看，Agent 的内部到底发生了哪些变化。

---

## 第 10 页：六个核心技术概念

预计时间：2 分钟

### 主讲稿

如果从宏观上看，今天的 Agent 和 2023 年的 Agent 好像没差太多，还是 Prompt、Planning、Memory、Tools 这些模块。

但真正的变化在于：模块名字没变，模块的实现方式变了。

Prompt 不再是一个巨大 System Prompt，而是变成上下文工程。  
Planning 不再只是 CoT，而是复杂任务规划。  
Memory 不再只是向量检索，而是短期压缩加长期文件系统沉淀。  
Tools 不再只是 Function Call，而是 CLI、脚本和 MCP 等多种方式。  
Workflow 不再只是外部编排，也可以内化成 Skill。  
Environment 不再可有可无，而是 Agent 真正执行任务的基础设施。

这一页可以作为后半场的地图。接下来我们逐个拆。

### 过渡句

先从最熟悉、也最容易被误解的 Prompt 开始。

---

## 第 11 页：Prompt，从深耦合“小作文”到渐进式加载

预计时间：5 分钟

### 主讲稿

早期做 Agent，很多人的第一反应是写 Prompt。

而且是越写越长。一个 System Prompt 里可能包括：你是谁、你的任务是什么、你要遵守哪些规则、你有哪些工具、输出格式是什么、不要做什么、遇到异常怎么办、给几个例子。

这在小场景里没问题，但场景一多就会失控。

比如你做一个内容创作 Agent，写文章有一套 Prompt，改标题有一套 Prompt，做配图有一套 Prompt，做小红书风格有一套 Prompt，做技术文章又一套 Prompt。最后维护的不是系统，而是一堆“小作文”。

现在的趋势是把 Prompt 解耦。

System Prompt 只保留最稳定、最底层的行为规范，比如安全边界、工具使用原则、沟通风格。具体任务方法放到 Skill，用户偏好放到配置文件，领域知识放到文档或知识库。Agent 需要什么，就渐进式加载什么。

这叫上下文的动静分离。

### 可展开例子

可以用“公司制度”和“项目资料”的类比：

System Prompt 像公司基本制度，比如不能泄露客户数据、不能乱删生产数据、遇到危险操作要确认。这些规则很稳定。

Skill 像岗位 SOP，比如怎么写周报、怎么处理客户投诉、怎么发布版本。这些是具体任务方法。

Memory 像个人工作习惯，比如老板喜欢一页纸摘要，客户 A 不喜欢太技术化表达。

Domain Docs 像项目资料，比如这个系统的业务架构、接口文档、领域词汇。

如果你把这些全塞进一个 System Prompt，就像把公司制度、员工手册、客户偏好、项目文档全部写在一张纸上，当然会乱。

### 技术补充

为什么渐进式加载重要？

第一，节省上下文。模型上下文不是无限的，把无关内容塞进去会挤占注意力。

第二，提高可维护性。业务规则变了，只改对应文件，不需要重写整个 Prompt。

第三，提高组合能力。同一个 Agent 可以根据任务加载不同 Skill，而不是每个任务创建一个新 Agent。

### 过渡句

Prompt 解决的是“告诉模型什么”，Planning 解决的是“模型接下来怎么做”。

---

## 第 12 页：Planning，从“想一步”到“规划一段路”

预计时间：5 分钟

### 主讲稿

早期 Planning 很多时候其实就是 CoT，也就是让模型一步一步想。

这在简单问题上有效。比如算一道数学题，或者分析一个短逻辑问题，模型一步步推理就能得到答案。

但真实任务不是一道题，而是一段路。

比如“帮我重构这个模块”。这不是一步推理能解决的。它需要先理解现状，再判断风险，再拆分改动，再修改代码，再验证，再处理边界情况。

所以今天的 Planning 更像项目管理。Agent 要能生成 Todo List，要能更新状态，要能在执行中调整计划。

更进一步，一些复杂系统还会引入 Sub-agent。主 Agent 负责总体规划，把搜索、测试、代码审查、文档整理等任务交给不同子 Agent。这个时候 Planning 已经从“脑内思考”变成了“组织协作”。

### 可展开例子

可以用“装修房子”的例子：

如果你问：“墙面刷什么颜色好？”这是一问一答。

但如果你说：“帮我完成一次旧房翻新。”这就是长程任务。你要先测量房屋，确定预算，找设计方案，安排水电，买材料，施工验收。

如果没有 Planning，工人可能今天刷墙，明天发现水电还没改，后天又把墙砸了重来。

Agent 也是一样。没有规划的 Agent 很容易先做了容易的事，但忘记关键依赖。

再举代码例子：

用户说：“把登录方式从密码改成钉钉登录。”

一个好的 Agent 不能直接开始改按钮。它要先看 SPEC，确认认证流程；再看前端登录页；再看后端鉴权；再看 SDK；再看环境变量；再看测试；最后才动手。

### 可展开观点

Planning 的难点不是生成一个漂亮计划，而是计划能不能随着现实变化。

很多任务开始时信息不完整。Agent 只有做了一步，才发现下一步该怎么做。比如改代码时，只有跑了测试才知道哪里失败；查资料时，只有读了第一批资料才知道哪些关键词更有效。

所以好的 Planning 是动态的，不是一次性写死的。

### 过渡句

长程任务一旦变长，就一定会遇到上下文问题。这就引出 Memory 的变化。

---

## 第 13 页：Memory，从检索增强到文件系统化沉淀

预计时间：6 分钟

### 主讲稿

Memory 早期最常见的理解是 RAG，也就是把外部知识放进向量数据库，需要时检索出来。

这个思路非常重要，尤其适合大量文档知识。但随着 Agent 任务变复杂，大家发现 Memory 不只是知识检索。

Memory 至少分两类。

第一类是短期记忆，也就是当前任务上下文。问题是上下文窗口有限，任务一长，对话和工具结果会越来越多。不能无限塞进去，所以需要压缩、摘要、提取重点。

第二类是长期记忆，也就是跨任务沉淀。它包括用户偏好、项目规则、历史经验、领域知识。

现在一个明显趋势是：长期记忆越来越多回到文件系统。比如 `MEMORY.md`、每日日志、项目文档、Obsidian 知识库、Skill 文件。这些东西不只是给模型看，也给人看。可读、可改、可审查。

当然，这不代表向量数据库没用了。大规模企业知识库仍然需要检索。但更成熟的方式往往是混合：文件系统负责组织和显式结构，向量检索负责大规模召回。

### 可展开例子

可以讲“人类记忆”的类比：

短期记忆像你开会时脑子里记着刚刚谁说了什么。会议一长，你不可能记住每一句话，所以你会记重点。

长期记忆像你的笔记本和知识库。你不会把所有知识都背在脑子里，而是会写成文档、目录、标签。

RAG 像搜索引擎。你需要什么，就搜索相关资料。

文件系统 Memory 像整理好的书架。你知道某类资料在哪个文件夹，哪个文档是规范，哪个文档是经验总结。

两者结合才好用。只有搜索，没有组织，会召回很多碎片；只有组织，没有检索，资料多了也找不到。

### 可展开案例

项目型 Agent 的 Memory 可以这样设计：

`AGENTS.md` 存项目协作规则。  
`SPEC/` 存产品和接口规范。  
`docs/domain.md` 存领域词汇。  
`MEMORY.md` 存用户偏好和反复出现的经验。  
`skills/` 存可复用任务流程。  
必要时再加 SQLite 或向量索引做检索。

这样 Agent 不是每次都从零开始读世界，而是在一个结构化环境里工作。

### 风险补充

Memory 最大的问题是污染。

如果 Agent 记了过时规则，下次会误用。  
如果 Agent 把临时偏好当永久规则，会越用越怪。  
如果 Memory 没有来源和更新时间，人也不知道该不该相信。

所以长期 Memory 最好是可读文件，并且有清晰边界：什么能记，什么不能记，什么时候更新，谁来审查。

### 过渡句

有了记忆还不够，Agent 真正做事必须使用工具。Tools 是这几年变化最大的模块之一。

---

## 第 14 页：Tools，从 Function Call 到 CLI / Script

预计时间：6 分钟

### 主讲稿

早期 Agent 工具调用的主流方式是 Function Call。

你把一个能力封装成函数，告诉模型函数名、参数、描述。模型需要时调用函数。这是非常标准、可控的方式。

但 Function Call 有一个现实问题：每个工具都要人为适配模型。

你要写 API，要写 Schema，要写参数说明，要处理鉴权，要维护版本。工具一多，管理成本非常高。

而现在很多 Agent，尤其是 Coding Agent，越来越依赖 CLI 和 Script。

为什么 CLI 对 Agent 友好？因为大模型在预训练中看过大量命令行、代码、文档。`git status`、`rg`、`npm test`、`python script.py` 这些对模型来说是熟悉的语言。

对人来说，CLI 很不友好，因为参数多、记不住。但对模型来说，CLI 反而是天然接口。它可以读 `--help`，可以组合命令，可以写脚本。

Script 则进一步把复杂逻辑封装起来。比如一个 Python 脚本内部处理鉴权、分页、重试、格式化，Agent 只需要传入核心参数。

这就是从“人为适配模型”转向“利用模型原生能力”。

### 可展开例子

可以举“查文件”的例子：

Function Call 方式可能要提供：

`search_files(query: string, path: string, fileType?: string)`  
`read_file(path: string, offset?: number, limit?: number)`  
`write_file(path: string, content: string)`  

每个函数都要描述清楚。

CLI 方式则是：模型知道可以用 `rg` 搜索，用文件读取工具读文件，用脚本处理数据。它不一定需要每个操作都有一个专门 API。

再举“批量处理 Excel”的例子：

如果全用 Function Call，你可能要设计很多接口：读取 Excel、筛选行、转换格式、写入新文件。

但如果给 Agent 一个 Python 环境，它可以用 pandas 写一个脚本，直接完成复杂转换。这个灵活性非常高。

### MCP 补充

可以顺带讲 MCP。

MCP 的价值是让工具注册和发现更标准。它解决的是“工具怎么暴露给 Agent”的问题。

但 MCP 不代表所有工具都必须是远程 API。很多时候 MCP 后面也可以包装本地脚本、CLI、文件资源。

所以我们可以这样理解：

Function Call 是工具接口。  
MCP 是工具协议。  
CLI 是模型天然会用的操作界面。  
Script 是可复用的执行封装。  

它们不是互斥关系。

### 风险补充

CLI 和 Script 也带来风险。

权限太大可能误删文件。  
脚本写错可能批量破坏数据。  
命令执行需要隔离和审计。  

所以 Tools 的演进必须和 Environment 一起看。工具越强，运行环境越要安全。

### 过渡句

工具解决了“怎么做”，Workflow 解决的是“做事路径如何被约束和复用”。

---

## 第 15 页：Workflow，从刚性编排到动态混合封装

预计时间：5 分钟

### 主讲稿

Workflow 早期是外部编排：第一步做什么，第二步做什么，第三步做什么，都在流程引擎里写死。

这很适合稳定流程，但不适合变化很快的复杂任务。

现在出现了一个趋势：很多 Workflow 的逻辑开始内化到 Skill 里。

Skill 是什么？它不只是 Prompt。一个好的 Skill 通常包括任务说明、适用场景、操作步骤、注意事项、示例，有时还会带脚本资源。

这意味着，一个可复用流程不一定非要画在 Workflow 引擎里，也可以写成 Markdown 加脚本。

但这不代表 Workflow 消失了。当前更合理的方式是混合架构：

灵活、可变、依赖模型判断的部分，用 Skill。  
高风险、强合规、必须稳定的主流程，用 Workflow。  
重复、精确、可程序化的步骤，用 Script。  

### 可展开例子

可以讲“报销系统”和“差旅建议”的对比：

报销审批适合 Workflow：提交发票、校验金额、匹配预算、主管审批、财务复核、打款。每一步都必须固定。

但“帮我规划一次最省时间的出差”更适合 Agent Skill。它需要综合航班、酒店、会议时间、个人偏好、天气、预算。这里有很多动态判断。

如果最终要提交差旅申请，再进入 Workflow。

这就是混合：前面自主规划，后面固定审批。

### 可展开观点

Skill 和 Workflow 的区别可以这样讲：

Workflow 像地图上画好的轨道。火车只能沿轨道跑，稳定但不灵活。  
Skill 像给司机的一本驾驶手册。司机知道原则和方法，可以根据路况调整。  
Script 像自动驾驶里的某个固定能力，比如自动泊车。执行精确，但范围明确。

成熟系统要把三者组合起来。

### 过渡句

当 Agent 开始使用工具、执行脚本、读写文件，它就不再是一个无状态问答系统了。它需要环境。

---

## 第 16 页：Environment，从无状态调用到有状态运行时

预计时间：5 分钟

### 主讲稿

早期 Agent 基本是无状态的。你发一个请求，它返回一个结果。最多保留一点对话上下文。

但今天的 Agent 如果要做长程任务，就必须有运行环境。

为什么？因为它要读文件、写文件、生成中间结果、运行命令、保存日志、管理 Memory 和 Skill。

一个没有 Workspace 的 Agent，就像一个没有办公桌的员工。每次工作都只能临时记在脑子里，做完东西也没地方放。

运行环境主要有两类：

第一类是本地电脑。优点是便利，Agent 可以直接帮你整理文件、改项目、操作本地工具。缺点是风险高，权限边界必须清楚。

第二类是沙箱或云服务器。优点是隔离、安全、可控，适合企业生产。缺点是对本地环境和个人应用的访问不如本地灵活。

### 可展开例子

可以讲“员工办公环境”的类比：

一个员工要高效工作，需要电脑、文件夹、公司系统账号、权限、工作台、历史文档。

Agent 也一样。它需要：

Workspace：放代码和生成物。  
Shell：执行命令。  
File System：读写文档。  
Network：访问外部资源。  
Sandbox：隔离风险。  
Logs：记录做过什么。  
Permissions：限制能做什么。  

如果没有这些，它就只能停留在聊天层面。

### 安全补充

Environment 的核心问题是权限。

如果你让 Agent 在本地电脑上拥有全部权限，它理论上可以删除文件、提交代码、访问敏感信息。这不是模型善不善良的问题，而是系统设计不能假设模型永远不犯错。

所以企业级 Agent 一定需要：

最小权限原则。  
危险操作确认。  
沙箱隔离。  
操作日志。  
可回滚机制。  

工具越强，环境越重要。

### 过渡句

到这里，六个模块都讲完了。我们用一页把变化串起来。

## 第 17 页：六个概念的范式变化

预计时间：3 分钟

### 主讲稿

这一页是整场分享的中间总结。

Prompt 从深耦合到渐进式加载，说明我们不再把所有知识塞进一个大 Prompt，而是开始做上下文工程。

Planning 从思维链到长程任务规划，说明 Agent 不再只是想一步，而是要规划一段路。

Memory 从 RAG 到文件系统加检索混合，说明记忆不只是召回知识，也包括可读、可维护的经验沉淀。

Tools 从 Function Call 到 CLI / Script，说明我们开始利用模型原生的代码和命令行能力。

Workflow 从刚性编排到 Skill 加 Workflow 混合，说明流程不再只有外部引擎一种形态。

Environment 从无状态到 Runtime / Workspace，说明 Agent 真正开始像一个执行者，需要工作环境。

这些变化合在一起，意味着 Agent 从“模型应用”变成了“工程系统”。

### 可展开观点

可以强调一个判断：

如果一个 Agent 系统只有 Prompt，没有工具和环境，它大概率只是聊天增强。  
如果它有工具，但没有规划和校验，它可能能做事，但不稳定。  
如果它有规划和工具，但没有 Memory，它每次都从零开始。  
如果它有 Memory，但没有权限和环境隔离，它可能有安全风险。  

所以高质量 Agent 不是单点能力强，而是系统组合合理。

### 过渡句

那么落到实际项目里，我们到底该怎么选？下一页给一个简单判断方法。

---

## 第 18 页：怎么选，不看概念新不新，看任务像什么

预计时间：5 分钟

### 主讲稿

技术选型最怕的是追新概念。

看到自主 Agent 火，就什么都做成自主 Agent；看到 Workflow 稳，就什么都画流程；看到 Memory 热，就什么都加向量库。这样很容易过度设计。

我建议按任务形态来选。

如果任务规则明确、流程稳定、容错率低，优先 Workflow。比如审批流、固定表单处理、财务核对、数据同步。

如果任务目标复杂、路径不确定、需要搜索和判断，引入自主 Agent。比如代码修改、问题排查、行业研究、竞品分析、客户洞察。

如果任务长期复用、涉及用户偏好和组织经验，就建设 Memory 和 Skill。比如团队代码规范、写作风格、项目知识库、常见故障处理流程。

这三个方向可以组合，不需要二选一。

### 可展开案例 1：企业客服

企业客服系统可以这样设计：

常见问题回答：RAG 加简单工具调用。  
退款审批：Workflow。  
复杂投诉处理：自主 Agent 先整理事实、判断情绪和风险，再建议处理方案。  
处理经验：沉淀成 Skill，比如“高价值客户投诉处理流程”。  
客户偏好和历史：沉淀到 Memory。

### 可展开案例 2：软件研发

软件研发 Agent 可以这样设计：

简单问答：解释代码、总结模块。  
明确流程：提交前检查、生成 changelog、创建 PR。  
复杂任务：自主 Agent 读代码、拆任务、修改、验证。  
团队规则：放到 AGENTS.md 或 SPEC。  
常见任务：沉淀成 Skill，比如“修复前端表单 bug 的流程”。  
执行环境：本地 workspace 或云端沙箱。

### 可展开案例 3：内容创作

内容创作 Agent 可以这样设计：

选题收集：自主搜索和整理。  
写作风格：Memory。  
文章结构：Skill。  
发布检查：Workflow，比如标题、摘要、封面、标签、敏感词检查。  
素材库：文件系统加检索。

### 过渡句

选型讲完后，我们回到一个更大的结论：Agent 的外形好像没变，但内核已经变了。

## 第 19 页：Agent 的“形”没变，但“神”已经变了

预计时间：4 分钟

### 主讲稿

如果我们只看模块名，今天的 Agent 和早期理论框架差不多，还是 Prompt、Planning、Memory、Tools。

但“形”没变，“神”已经变了。

以前我们更像是在调模型：Prompt 怎么写得更聪明，CoT 怎么诱导得更好，工具描述怎么让模型理解。

现在我们更像是在建系统：上下文怎么组织，任务怎么规划，工具怎么封装，环境怎么隔离，经验怎么沉淀，错误怎么恢复。

这就是从“魔法调优”到“系统工程”的变化。

我认为未来一段时间，高质量 Agent 的核心思想就是：

用工程化确定性，承载模型的不确定性。

模型负责推理、生成、判断、创造。  
工程负责边界、流程、权限、环境、验证。  

两者结合，Agent 才能真正解决复杂现实问题。

### 可展开例子

可以用“自动驾驶”的类比：

自动驾驶不是只靠一个聪明模型。它需要传感器、地图、规则、控制系统、安全冗余、日志、人工接管机制。

Agent 也是一样。不能只说“模型越来越强，所以工程不重要”。恰恰相反，模型越能做事，工程边界越重要。

因为一个只会回答问题的模型，犯错影响有限。  
一个能执行命令、改代码、发邮件、调用系统的 Agent，犯错影响就大得多。  

所以能力越强，越需要系统工程。

### 过渡句

最后用一页收束：Agent 正在从工具变成伙伴，但这个伙伴需要被正确设计。

## 第 20 页：Agent 正在从工具变成伙伴

预计时间：4 分钟

### 主讲稿

最后总结一下。

Agent 这几年最大的变化，不只是模型能力更强，也不是框架名字更多，而是它的角色变了。

它从一个回答问题的工具，逐步变成一个可以执行任务的系统；又从一个一次性执行系统，开始走向可以沉淀经验、长期成长的数字伙伴。

但我们也要保持清醒。Agent 不是万能的。它适合处理开放性、复杂性、需要工具协作的任务；但对于强规则、低容错、强合规的场景，仍然需要 Workflow、权限、审计和人工确认。

所以我们构建 Agent 时，不应该盲目追求“全自动”，而应该追求“在合适边界内自动化”。

一个好的 Agent 系统，应该做到三件事：

第一，理解演化。知道不同范式解决什么问题。  
第二，按场景选型。不要为了新架构而新架构。  
第三，沉淀资产。把 Prompt、Skill、Memory、Workflow、工具脚本变成可复用的组织能力。

如果用一句话结尾，我会说：

未来的 Agent，不只是更聪明的模型，而是被工程化组织起来的智能执行系统。

### 结束互动问题

可以抛给听众几个问题：

你现在的业务场景，更适合 Workflow，还是自主 Agent？  
你们团队有没有把经验沉淀成 Skill 或文档？  
你们现在的 Agent 工具有没有安全边界和运行环境？  
如果让 Agent 长期服务一个团队，它应该记住什么，又不应该记住什么？

这些问题比“用哪个框架”更重要。

## 可选扩展段落：如果现场时间还多

### 扩展 1：Agent 和自动化脚本的区别

很多人会问，Agent 和传统自动化脚本有什么区别？

脚本适合确定性任务。输入明确、步骤明确、输出明确。比如每天 9 点拉取数据、生成报表、发邮件。

Agent 适合不确定性任务。目标明确，但路径不固定。比如“分析这周销售下降的原因”。它可能要查数据、看活动、看竞品、看客户反馈，还要做判断。

所以 Agent 不是替代所有脚本，而是补上脚本不擅长的开放式判断。最好的方式是让 Agent 调用脚本，而不是让 Agent 用自然语言硬做所有事。

### 扩展 2：为什么 Coding Agent 发展最快

Coding Agent 是 Agent 领域进展最快的方向之一，因为代码世界对模型非常友好。

第一，训练数据多。模型见过大量代码、命令、错误日志和技术文档。

第二，反馈明确。代码能不能跑、测试过不过、类型检查有没有错误，都是相对客观的反馈。

第三，工具成熟。Git、Shell、测试框架、包管理器、CI 都是天然工具。

第四，产物可审查。代码 diff 可以被人 review，不像某些业务决策很难判断对错。

所以 Coding Agent 是自主 Agent 的最佳试验场。

### 扩展 3：为什么企业落地不能只看 Demo

Demo 里 Agent 很容易显得厉害，因为环境干净、任务单一、失败成本低。

企业落地不一样。真实环境里有权限、历史数据、异常流程、合规要求、系统依赖、多人协作、版本管理。

所以评估 Agent 时，不能只看它“能不能成功一次”，还要看：

失败时能不能恢复？  
过程能不能追踪？  
权限能不能控制？  
结果能不能验证？  
经验能不能沉淀？  
成本能不能接受？

这些问题决定了它能不能从 Demo 走向生产。

### 扩展 4：Agent 的未来可能是什么

未来 Agent 可能会往几个方向发展。

第一，更强的多模态能力。Agent 不只读文字和代码，也能看界面、看图片、看视频、听语音。

第二，更标准的工具生态。MCP 或类似协议会让工具接入更方便。

第三，更成熟的运行环境。云端沙箱、本地权限控制、审计日志会成为标配。

第四，更强的长期记忆。Agent 会越来越像一个长期协作对象，而不是一次性问答工具。

第五，更细粒度的人机协作。不是全自动替代人，而是在关键节点请求人确认，把人放在决策链路里。
