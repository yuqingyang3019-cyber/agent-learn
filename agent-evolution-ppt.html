总览

Agent 发展的四个阶段

2023

早期 Agent

被动式 ReAct，一问一答，适合短链路任务。

2024

工作流 Agent

用流程约束模型，强调结构化、可控性和稳定下限。

2025

自主 Agent

能规划、能迭代、能处理更长程的复杂任务。

2026

自进化 Agent

沉淀 Skill、Memory 和知识，追求越用越好用。

阶段一

早期 Agent：增强版 Chatbot

核心形态是 ReAct：模型先想一下，再调用工具或回答。它打开了想象力，但还不适合长链路复杂任务。

用户提问 → Reasoning → Observe → Response

像什么？

像一个会查工具的聊天助手。

能做什么？

单点、短链路、小任务。

怕什么？

上下文变长、任务变复杂、推理轮次过多。

阶段二

工作流 Agent：先把路修好，再让模型跑

当模型不够稳时，工程上最有效的办法是把关键路径固定下来。

动图预留 5 秒：模型像小车一样沿固定轨道通过 1、2、3、4 个节点

架构特征

固定流程 + 关键节点嵌入 LLM，或者 LLM 调用预定义子流程。

典型价值

牺牲一些灵活性，换来更高的可控性和可解释性。

适合场景

企业流程、重复性工作、结果稳定性要求高的业务。

阶段三

自主 Agent：从“帮你答”到“帮你做”

以 Manus、Claude Code、Codex 等为代表，Agent 开始具备复杂规划和长程执行能力。

模糊目标 → 拆任务 → 调工具 → 校验迭代 → 交付结果

复杂 Planning

能把大目标拆成可执行的 Todo List。

长程任务

能连续运行较长时间，处理代码库或复杂流程。

自我校验

执行过程中发现问题、修正错误、再次验证。

阶段四

自进化 Agent：做完任务，还把经验留下来

新一代 Agent 不只是消耗上下文，而是把经验沉淀成 Skill、Memory、知识库，甚至反馈到训练。

动图预留 6 秒：任务完成后生成 Skill.md、Memory.md、知识卡片，汇入大脑

机制

通过记忆、反馈循环和反思，把教训转化为策略。

目标

越用越好用，减少重复指导和重复踩坑。

意义

Agent 从一次性工具，变成可积累的数字资产。

重要提醒

四个阶段不是替代关系，而是工具箱关系

稳

流程清楚

用 Workflow，先保证下限和可解释性。

活

问题复杂

用自主 Agent，让它规划、搜索、执行、校验。

长

长期使用

加 Memory 和 Skill，让经验可沉淀、可复用。

这里可以强调：不要为了新架构而新架构，选型要看场景复杂度、稳定性要求和成本。

Prompt

从深耦合“小作文”，到渐进式加载

早期做法

一个任务，一个大 Prompt

  • 人设、目标、约束、示例全塞进 System Prompt。
  • 场景一多，维护成本迅速上升。
  • Prompt 变成难以管理的“小作文仓库”。
现在趋势

稳定系统指令 + 外部上下文

  • System Prompt 保持稳定、通用、简洁。
  • 任务方法沉淀到 SKILL.md。
  • 偏好、规范、领域知识放入配置文件，按需读取。
动图预留 4 秒：一个巨大的 Prompt 被拆成 System、Skill、Memory、Domain Docs 四个模块。
Planning

从“想一步”，到“规划一段路”

早期

CoT 思维链

依赖类似 “Let's think step by step” 的提示词，让模型线性推理。

现在

复杂长程任务规划

能拆任务、排顺序、动态调整，并在必要时调用子 Agent 协作。

结构化分解

把宏大目标拆成子任务。

多步协同

按计划执行，并根据反馈改计划。

子 Agent

把专项工作交给更合适的执行单元。

Memory

从检索增强，到文件系统化沉淀

短期记忆

重点从“存储”变成“压缩”

  • 基于 token 阈值触发压缩。
  • 把中间过程总结成结构化摘要。
  • 提取关键事实，减少无关噪音。
长期记忆

从纯向量库走向混合管理

  • 事项型记忆写入 MEMORY.md 或日志。
  • 知识型记忆沉淀到 Markdown 知识库。
  • 大规模场景再搭配向量检索或 SQLite。
通俗讲法：短期记忆像“会议纪要”，长期记忆像“团队知识库”。
Tools

变化最大的地方：从 Function Call 到 CLI / Script

Function Call 时代

人为适配模型

  • 每个能力都要封装 API。
  • 要维护名称、描述、参数、返回结构。
  • 工具数量越多，Schema 越重。
CLI / Script 时代

利用模型原生能力

  • 模型天然熟悉命令行、代码和技术文档。
  • 第三方 CLI 可通过 --help 即时学习。
  • 复杂逻辑封装为脚本，Agent 只关心怎么调用。
动图预留 5 秒：一堆 API Schema 收缩成一个命令行窗口和几个脚本文件。
Workflow

从刚性编排，到动态混合封装

旧

外部流程引擎

步骤固定，状态跳转明确，适合保证执行下限。

新

Skill 封装

把任务逻辑、约束、示例写进 Skill 文档，脚本负责精确执行。

合

混合架构

标准子任务用 Skill，关键主干流程仍保留 Workflow 或封装为 Tool。

一句话：Skill 提供灵活性，Workflow 保住确定性。
Environment

从无状态调用，到有状态运行时

早期

像一次请求

问一句、答一句，工具调用基本无状态，不需要专属工作空间。

现在

像一个数字员工

要读写文件、生成中间产物、管理日志、维护 Skill 和 Memory。

本地电脑

便利、灵活，但需要权限控制和用户确认，避免误删误改。

沙箱 / 云服务器

企业级主流选择，用隔离环境控制风险和资源边界。

选型建议

怎么选，不看概念新不新，看任务像什么

规则明确

优先 Workflow

例如审批流、表单处理、固定数据清洗。重点是稳定、可追踪、可解释。

目标复杂

引入自主 Agent

例如代码修改、调研分析、跨系统排障。重点是规划、工具使用和自检。

长期复用

建设 Memory / Skill

例如团队知识、个人偏好、项目经验。重点是沉淀资产，减少重复沟通。

核心观点

Agent 的“形”没变，但“神”已经变了

今天的 Agent 依然有 Prompt、Planning、Memory、Tools，但每个模块的组织方式、数据流和工程边界都已经重构。 变化的本质：用工程化确定性，承载模型的不确定性。

不迷信模型智商

通过文件化、脚本化、沙箱化，让模型更稳定地发挥优势。

不盲追最新架构

根据复杂度、稳定性、成本选择合适范式，必要时组合使用。

结束页

Agent 正在从工具
变成伙伴

未来还会继续变化，但高质量 Agent 的底层方法不会变：让模型负责推理与创造，让工程负责边界与确定性。
